1. Visão Geral

O que é o processo de Projeção CIF?

O módulo projecao_cif é responsável por projetar valores CIF (Cost, Insurance, Freight) para indicadores do Brasil que não possuem esse valor calculado. Este processo utiliza dados históricos para calcular uma razão média (alpha) entre CIF e FOB por mês, e então aplica essa razão para calcular valores CIF faltantes com base na projeção do FOB.

Funcionalidade principal: O processo calcula valores CIF faltantes multiplicando a projeção do FOB pela razão histórica média (alpha) entre CIF e FOB, permitindo preencher lacunas onde apenas valores FOB estão disponíveis.

⚠️ IMPORTANTE: Execução no Airflow

Este processo está configurado e roda na DAG do Airflow chamada costdriver_db_ETL. Esta DAG possui conexão configurada ao banco de dados de produção, permitindo que o processo execute diretamente no ambiente produtivo.

O processo deve ser executado após qualquer atualização ou modificação nos dados do Brasil para garantir que os valores CIF faltantes sejam calculados corretamente com base na projeção do FOB.

Objetivo Principal

Calcular valores CIF faltantes em indicadores de importação do Brasil (paisID = 27) que possuem apenas valores FOB projetados, utilizando uma metodologia baseada em médias históricas mensais da razão CIF/FOB (alpha).

Como funciona: O processo utiliza a projeção do FOB como base e multiplica pela razão histórica (alpha) para obter a projeção do CIF, permitindo preencher valores CIF que estão faltando nos dados.

Características do Processo

  • Processo SQL puro: Implementado inteiramente em SQL Server
  • Foco no Brasil: Processa apenas dados de importação do Brasil (paisID = 27)
  • Baseado em histórico: Usa dados históricos desde 2022-01-01 para calcular médias
  • Projeção mensal: Calcula alpha médio por mês para cada indicador
  • Validação antes de atualizar: Script inclui SELECT para validação antes do UPDATE

2. Objetivo e Contexto

Por que existe este processo?

Alguns indicadores de importação do Brasil possuem apenas valores FOB (Free On Board) projetados, mas não possuem valores CIF (Cost, Insurance, Freight). O processo de projeção CIF calcula valores CIF faltantes baseando-se em padrões históricos, utilizando a projeção do FOB como base para calcular a projeção do CIF.

Uso principal: Este processo pode ser usado para calcular CIF faltante sempre que houver valores FOB projetados mas valores CIF ausentes, garantindo que os dados estejam completos para análises e cálculos downstream.

Quando os valores CIF estão faltando?

Valores CIF podem estar faltando quando:

  • Dados são coletados de fontes que não fornecem informações de frete e seguro
  • Novos indicadores são adicionados sem histórico completo
  • Dados são atualizados e alguns valores CIF não são recalculados
  • Ocorrem mudanças na estrutura de dados que afetam o cálculo de CIF

Metodologia Utilizada

O processo utiliza a seguinte abordagem:

  1. Identifica indicadores elegíveis: Indicadores de importação do Brasil que precisam de projeção CIF
  2. Calcula alpha histórico: Razão CIF/FOB a partir de dados históricos (Base = 0)
  3. Agrega por mês: Calcula média de alpha por mês para cada indicador
  4. Projeta valores: Aplica alpha médio mensal à projeção do FOB para calcular a projeção do CIF (Base = 2)
  5. Atualiza banco: Atualiza valores CIF na tabela tblindiceprincipal_mes

4. Metodologia de Projeção

Conceito de Alpha (α)

Alpha é a razão entre Valor CIF e Valor FOB: 

alpha = Valor_CIF / Valor_FOB

Esta razão representa quanto o valor CIF é maior que o valor FOB, incluindo custos de frete e seguro.

Exemplo: Se FOB = 1000 e CIF = 1100, então alpha = 1.1 (CIF é 10% maior que FOB)

Como Funciona: Projeção do CIF com Base na Projeção do FOB

O processo utiliza a seguinte abordagem para calcular valores CIF faltantes:

  1. Base de cálculo: Utiliza a projeção do FOB (Valor_FOB projetado) como base
  2. Razão histórica: Calcula alpha médio mensal a partir de dados históricos (razão CIF/FOB)
  3. Cálculo do CIF: Multiplica a projeção do FOB pelo alpha médio mensal para obter a projeção do CIF
  4. Resultado: Obtém valores CIF faltantes calculados com base na projeção do FOB
Projeção_CIF = Projeção_FOB × alpha_médio_mensal

Uso principal: Este processo pode ser usado para calcular CIF faltante sempre que houver projeção do FOB disponível mas valores CIF ausentes, garantindo que os dados estejam completos para análises e cálculos downstream.

Etapas da Metodologia

CTE 1: filtered_ids_import

Objetivo: Identificar todos os indicadores de importação do Brasil

SELECT indiceprincipalID
FROM tblsnapshot_indiceprincipal
WHERE paisID = 27          -- Brasil
  AND importexport = 1     -- Importação
  AND NCM IS NOT NULL      -- Com NCM válido

CTE 2: filtered_ids_import_needed

Objetivo: Identificar indicadores que precisam de projeção CIF

SELECT DISTINCT(indiceprincipalID)
FROM tblindiceprincipal_mes
WHERE indiceprincipalID IN (filtered_ids_import)
  AND Base = 2              -- Dados que precisam de projeção
  AND Valor_CIF IS NULL      -- Sem valor CIF

CTE 3: filtered_elegibles

Objetivo: Obter dados históricos e calcular alpha para cada registro

SELECT 
    indiceprincipalID,
    DataIndice,
    Base,
    Valor,                    -- Valor FOB
    Valor_cif,                -- Valor CIF
    Valor_CIF / NULLIF(Valor, 0) as alpha,  -- Razão CIF/FOB
    MONTH(DataIndice) as mes  -- Mês para agrupamento
FROM tblsnapshot_indiceprincipal_mes
WHERE DataIndice >= '2022-01-01'  -- Dados históricos

Observações:

  • Usa NULLIF(Valor, 0) para evitar divisão por zero
  • Extrai o mês da data para agrupamento sazonal
  • Inclui dados históricos (Base = 0) e dados que precisam de projeção (Base = 2)

CTE 4: alpha_month

Objetivo: Calcular média de alpha por mês para cada indicador

SELECT 
    IndicePrincipalID,
    mes,
    AVG(alpha) as alpha_mean_month
FROM filtered_elegibles
WHERE base = 0                    -- Apenas dados históricos
  AND Valor_CIF > Valor            -- Alpha > 1 (validação)
GROUP BY IndicePrincipalID, mes

Validações:

  • Apenas dados históricos (Base = 0) são usados para calcular a média
  • Apenas registros onde Valor_CIF > Valor (alpha > 1) são considerados
  • Agrupa por indicador e mês para capturar sazonalidade

CTE 5: forecast_data

Objetivo: Calcular valores CIF projetados

SELECT 
    A.indiceprincipalID,
    A.DataIndice,
    A.mes,
    A.Base,
    A.Valor,                              -- Valor FOB
    B.alpha_mean_month,                   -- Alpha médio do mês
    A.Valor * B.alpha_mean_month as Valor_Cif_forecast  -- Projeção CIF
FROM filtered_elegibles A
JOIN alpha_month B
    ON A.IndicePrincipalID = B.IndicePrincipalID
    AND A.mes = B.mes                     -- Match por mês
WHERE A.base = 2                          -- Apenas dados que precisam de projeção

Fórmula de Projeção:

Valor_CIF_Projetado = Valor_FOB_Projetado × alpha_médio_mensal

Uso: Esta fórmula permite calcular valores CIF faltantes com base na projeção do FOB, utilizando a razão histórica média entre CIF e FOB para cada mês.

Validação e Atualização

O script final faz um JOIN entre os dados projetados e a tabela de destino, identificando registros que precisam ser atualizados: 

SELECT *
FROM forecast_data A
JOIN tblindiceprincipal_mes B
    ON A.IndicePrincipalID = B.IndicePrincipalID
    AND A.DataIndice = B.DataIndice
WHERE (A.Valor_Cif_forecast <> B.Valor_CIF) 
   OR (B.Valor_CIF IS NULL AND A.Valor_Cif_forecast IS NOT NULL)

Condições para atualização:

  • Valor projetado é diferente do valor atual, OU
  • Valor atual é NULL e valor projetado não é NULL

5. Fluxo de Processamento

1. Identificar indicadores de importação do Brasil (paisID = 27, importexport = 1)
↓
2. Filtrar indicadores que precisam de projeção CIF (Base = 2, Valor_CIF IS NULL)
↓
3. Carregar dados históricos desde 2022-01-01 (Base = 0)
↓
4. Calcular alpha (razão CIF/FOB) para cada registro histórico
↓
5. Filtrar apenas alphas válidos (Valor_CIF > Valor, ou seja, alpha > 1)
↓
6. Agrupar por indicador e mês, calcular média de alpha
↓
7. Fazer JOIN dos dados que precisam de projeção com alpha médio mensal
↓
8. Calcular Valor_CIF_forecast = Valor_FOB_Projetado × alpha_médio_mensal (calcula CIF faltante)
↓
9. Validar dados projetados (SELECT para verificar antes de atualizar)
↓
10. Atualizar tabela tblindiceprincipal_mes com valores CIF projetados

Detalhamento das Fases

Fase 1-2: Identificação de Indicadores Elegíveis

O processo identifica quais indicadores precisam de projeção CIF:

  • Apenas indicadores de importação do Brasil (paisID = 27, importexport = 1)
  • Indicadores que têm Base = 2 (dados que precisam de projeção)
  • Indicadores onde Valor_CIF IS NULL (sem valor CIF)

Fase 3-5: Cálculo de Alpha Histórico

Para cada indicador elegível, o processo:

  • Carrega dados históricos desde 2022-01-01 (Base = 0)
  • Calcula alpha = Valor_CIF / Valor_FOB para cada registro
  • Filtra apenas registros válidos onde Valor_CIF > Valor (alpha > 1)
  • Isso garante que apenas razões válidas sejam usadas (CIF sempre maior ou igual a FOB)

Fase 6: Agregação Mensal

O processo agrega alpha por mês para capturar sazonalidade:

  • Agrupa por IndicePrincipalID e mes
  • Calcula AVG(alpha) para obter alpha médio mensal
  • Isso permite que diferentes meses tenham diferentes razões CIF/FOB
  • Exemplo: Janeiro pode ter alpha = 1.15, enquanto Dezembro pode ter alpha = 1.08

Fase 7-8: Projeção

Para cada registro que precisa de projeção CIF:

  • Faz JOIN com alpha médio do mesmo mês
  • Calcula a projeção do CIF usando a projeção do FOB: Valor_CIF_forecast = Valor_FOB_projetado × alpha_médio_mensal
  • Usa o mês do registro para encontrar o alpha correto (captura sazonalidade)
  • Resultado: Obtém valores CIF faltantes calculados com base na projeção do FOB

Importante: Este processo pode ser usado para calcular CIF faltante sempre que houver projeção do FOB disponível mas valores CIF ausentes, garantindo que os dados estejam completos.

Fase 9-10: Validação e Atualização

Antes de atualizar, o processo valida:

  • Compara valores projetados com valores atuais
  • Identifica apenas registros que realmente precisam ser atualizados
  • Atualiza apenas quando há diferença ou quando Valor_CIF é NULL

6. Execução no Airflow

⚠️ Processo Configurado no Airflow

Este processo está configurado e roda na DAG do Airflow chamada costdriver_db_ETL. Esta DAG possui conexão configurada ao banco de dados de produção, permitindo execução direta no ambiente produtivo.

Não é necessário executá-lo manualmente, a menos que haja necessidade de reprocessamento ou validação.

Configuração no Airflow

O processo é executado como uma task SQL na DAG costdriver_db_ETL:

  • DAG: costdriver_db_ETL
  • Tipo de Task: SQL Task (executa script SQL diretamente no banco)
  • Conexão: Configurada para banco de dados de produção
  • Banco de Dados: costdrivers (produção)
  • Script: NM/projecao_cif/processamento_BR.sql
  • Dependências: Deve ser executado após atualização dos dados do Brasil

⚠️ Conexão ao Banco de Produção

A DAG costdriver_db_ETL possui conexão configurada ao banco de dados de produção. Isso significa que:

  • O script SQL executa diretamente no ambiente produtivo
  • As atualizações são feitas na tabela tblindiceprincipal_mes em produção
  • É importante validar os dados antes de executar o UPDATE
  • Recomenda-se executar primeiro o SELECT para verificar os dados projetados

Estrutura do DAG

# DAG: costdriver_db_ETL
# Este DAG possui conexão ao banco de produção

@dag(
    dag_id="costdriver_db_ETL",
    schedule_interval=None,  # Execução manual ou após DAG do Brasil
    catchup=False
)
def costdriver_etl_dag():
    
    @task
    def executar_projecao_cif():
        # Executa script SQL no banco de produção
        execute_sql_file(
            'NM/projecao_cif/processamento_BR.sql',
            connection_id='costdriver_prod'  # Conexão ao banco de produção
        )
    
    projecao_cif_task = executar_projecao_cif()
    
dag_instance = costdriver_etl_dag()

Integração com DAG do Brasil

O processo de projeção CIF deve ser executado após o processamento dos dados do Brasil: 

# Fluxo no Airflow
COMEX_BRA_DAG >> costdriver_db_ETL  # Executa projeção CIF após dados do Brasil

Ordem de execução:

  1. DAG do Brasil executa: coleta → tratamento → cálculo (gera projeção do FOB)
  2. DAG costdriver_db_ETL executa após conclusão do DAG do Brasil
  3. Projeção CIF calcula valores CIF faltantes com base na projeção do FOB
  4. Valores CIF são atualizados na tabela de produção

7. Quando Executar

⚠️ CRÍTICO: Execução Após Atualizações do Brasil

Este processo DEVE ser executado após qualquer atualização ou modificação nos dados do Brasil.

Cenários que Requerem Execução

1. Atualização Regular de Dados

Após cada execução do pipeline do Brasil (OM/COMEX_BRA ou NM/globinho/BRA):

  • Novos dados podem ter sido inseridos sem valores CIF
  • Valores CIF podem precisar ser recalculados
  • Indicadores novos podem precisar de projeção

2. Modificação Manual de Dados

Após qualquer modificação manual nos dados do Brasil:

  • Correção de valores FOB
  • Inserção de novos registros
  • Atualização de indicadores existentes

3. Reprocessamento Completo

Quando há necessidade de reprocessar todos os dados:

  • Mudanças na metodologia de cálculo
  • Correção de bugs que afetaram valores CIF
  • Atualização de dados históricos

4. Validação e Auditoria

Para validar se os valores CIF estão corretos:

  • Executar o SELECT (sem UPDATE) para verificar projeções
  • Comparar valores projetados com valores existentes
  • Identificar indicadores que precisam de atenção

O que Acontece se Não Executar?

Consequências de não executar após atualizações:

  • Valores CIF podem ficar desatualizados ou NULL
  • Indicadores novos podem não ter valores CIF projetados
  • Cálculos downstream que dependem de CIF podem estar incorretos
  • Relatórios e análises podem mostrar dados incompletos

8. Detalhamento do Script SQL

Estrutura Completa do Script

USE costdrivers;

-- CTE 1: Identificar indicadores de importação do Brasil
WITH filtered_ids_import AS (
    SELECT indiceprincipalID
    FROM tblsnapshot_indiceprincipal WITH (NOLOCK)
    WHERE paisID = 27
      AND importexport = 1
      AND NCM IS NOT NULL
),

-- CTE 2: Identificar indicadores que precisam de projeção
filtered_ids_import_needed AS (
    SELECT DISTINCT(indiceprincipalID)
    FROM tblindiceprincipal_mes WITH (NOLOCK)
    WHERE indiceprincipalID IN (SELECT indiceprincipalID FROM filtered_ids_import)
      AND Base = 2
      AND Valor_CIF IS NULL
),

-- CTE 3: Obter dados históricos e calcular alpha
filtered_elegibles AS (
    SELECT
        B.indiceprincipalID,
        B.DataIndice,
        B.Base,
        B.Valor,
        B.Valor_cif,
        B.Valor_CIF / NULLIF(B.Valor, 0) as alpha,
        MONTH(B.DataIndice) as mes
    FROM filtered_ids_import_needed A
    JOIN (
        SELECT 
            indiceprincipalID,
            DataIndice,
            Base,
            Valor,
            Valor_cif
        FROM tblsnapshot_indiceprincipal_mes
        WHERE DataIndice >= '2022-01-01'
    ) B
        ON A.indiceprincipalID = B.indiceprincipalID
),

-- CTE 4: Calcular média de alpha por mês
alpha_month AS (
    SELECT 
        IndicePrincipalID,
        mes,
        AVG(alpha) as alpha_mean_month
    FROM filtered_elegibles
    WHERE base = 0
      AND Valor_CIF > Valor
    GROUP BY IndicePrincipalID, mes
),

-- CTE 5: Calcular valores CIF projetados
forecast_data AS (
    SELECT
        A.indiceprincipalID,
        A.DataIndice,
        A.mes,
        A.Base,
        A.Valor,
        B.alpha_mean_month,
        A.Valor * B.alpha_mean_month as Valor_Cif_forecast
    FROM filtered_elegibles A
    JOIN alpha_month B
        ON A.IndicePrincipalID = B.IndicePrincipalID
        AND A.mes = B.mes
    WHERE A.base = 2
)

-- Validação: SELECT para verificar antes de atualizar
SELECT *
FROM forecast_data A
JOIN tblindiceprincipal_mes B
    ON A.IndicePrincipalID = B.IndicePrincipalID
    AND A.DataIndice = B.DataIndice
WHERE (A.Valor_Cif_forecast <> B.Valor_CIF) 
   OR (B.Valor_CIF IS NULL AND A.Valor_Cif_forecast IS NOT NULL);

9. Troubleshooting

Problema: Nenhum registro encontrado para atualização

Sintoma: SELECT retorna 0 linhas

Possíveis causas:

  1. Todos os valores CIF já estão preenchidos
  2. Não há dados históricos suficientes para calcular alpha
  3. Filtros estão muito restritivos
  4. Dados do Brasil não foram atualizados recentemente

Solução:

  • Verifique se há dados em tblindiceprincipal_mes com Base = 2 e Valor_CIF IS NULL
  • Verifique se há dados históricos em tblsnapshot_indiceprincipal_mes desde 2022-01-01
  • Execute o SELECT de cada CTE separadamente para identificar onde está o problema

Problema: Alpha médio muito alto ou muito baixo

Sintoma: Valores CIF projetados parecem incorretos

Possíveis causas:

  1. Dados históricos com erros (CIF muito diferente de FOB)
  2. Poucos dados históricos para calcular média confiável
  3. Mudanças estruturais nos dados que afetam a razão CIF/FOB

Solução:

  • Valide os dados históricos antes de executar
  • Verifique se há outliers nos dados históricos
  • Considere aumentar o período histórico se necessário
  • Revise a validação Valor_CIF > Valor se necessário

Problema: Erro ao executar UPDATE

Sintoma: Erro de permissão ou constraint violation

Solução:

  1. Verifique se o usuário tem permissão UPDATE na tabela tblindiceprincipal_mes
  2. Verifique se há constraints que impedem a atualização
  3. Execute primeiro o SELECT para validar os dados
  4. Considere fazer UPDATE em batches menores se houver muitos registros

Problema: Processo não executa automaticamente no Airflow

Sintoma: DAG não é executado após atualização do Brasil

Solução:

  1. Verifique se o DAG está ativo no Airflow
  2. Verifique se as dependências estão configuradas corretamente
  3. Verifique os logs do Airflow para identificar erros
  4. Execute manualmente se necessário para validar

Dicas de Debug

  • Execute CTEs separadamente: Teste cada CTE individualmente para entender o fluxo
  • Valide antes de atualizar: Sempre execute o SELECT antes de descomentar o UPDATE
  • Verifique contagens: Conte registros em cada etapa para identificar onde há perda de dados
  • Analise alpha: Verifique se os valores de alpha fazem sentido (geralmente entre 1.0 e 1.5)
  • Compare com histórico: Compare valores projetados com valores históricos similares
  • Use transações: Envolva o UPDATE em uma transação para poder fazer rollback se necessário

Exemplo de Query de Validação

-- Verificar quantos indicadores precisam de projeção
SELECT COUNT(DISTINCT indiceprincipalID) as total_indicadores
FROM tblindiceprincipal_mes
WHERE paisID = 27
  AND importexport = 1
  AND Base = 2
  AND Valor_CIF IS NULL;

-- Verificar se há dados históricos suficientes
SELECT 
    indiceprincipalID,
    COUNT(*) as registros_historicos,
    MIN(DataIndice) as data_minima,
    MAX(DataIndice) as data_maxima
FROM tblsnapshot_indiceprincipal_mes
WHERE DataIndice >= '2022-01-01'
  AND Base = 0
  AND Valor_CIF IS NOT NULL
  AND Valor > 0
GROUP BY indiceprincipalID
HAVING COUNT(*) < 12  -- Menos de 12 meses de histórico
ORDER BY registros_historicos;
