1. Visão Geral

O que é o OM (Old Methodology)?

O módulo OM (Old Methodology - Metodologia Antiga) contém os processos de coleta, tratamento, cálculo e upload de dados de comércio exterior da metodologia antiga do sistema COMEX. Este módulo implementa uma arquitetura em camadas (medallion architecture) com três níveis de dados: Raw, Silver e Gold.

⚠️ IMPORTANTE: Estrutura de Dados OM/dados

O diretório OM/dados é onde estamos simulando a saída dos processos COMEX da metodologia antiga. Esta estrutura de dados representa os arquivos Parquet gerados pelos pipelines de cada país, organizados em três camadas:

  • raw/ - Dados brutos coletados das fontes externas
  • silver/ - Dados tratados, normalizados e validados
  • gold/ - Dados finais com cálculos e análises aplicadas

Quando o pipeline não está usando Azure Storage (use_azure=False), os dados são salvos localmente em OM/dados/{ISO_CODE}/{camada}/, simulando a estrutura de saída dos processos COMEX da metodologia antiga.

Objetivo Principal

Processar dados de comércio exterior de diferentes países através de um pipeline padronizado com 4 fases:

  1. Coleta (collect): Coleta dados brutos de fontes externas
  2. Tratamento (treat): Normaliza, valida e enriquece os dados
  3. Cálculo (calculate): Aplica cálculos estatísticos e suavização
  4. Upload (upload): Envia dados finais para a API de Cost Drivers

Características do Framework

  • Arquitetura orientada a objetos: Classe base abstrata com métodos específicos por país
  • Medallion Architecture: Dados organizados em Raw → Silver → Gold
  • Suporte a Azure e Local: Pode usar Azure ADLS2 ou storage local
  • Cache local: Sistema de cache para desenvolvimento (evita coletas repetidas)
  • Contrato de dados: Usa data-contract.yaml para mapeamento de colunas
  • Integração com Airflow: DAGs configurados para execução automatizada

3. Framework Base (ComexPipeline)

Classe ComexPipeline

A classe ComexPipeline é uma classe abstrata que define toda a lógica comum do pipeline. Ela utiliza o padrão Template Method, onde o fluxo geral é definido na classe base, mas métodos específicos são implementados pelas classes filhas de cada país.

Inicialização

def __init__(
    self, 
    iso_code: str, 
    start_date: Optional[datetime] = None,
    data_contract_path: str = None,
    ids_table_path: str = None,
    storage_base_path: str = "staging/comex",
    use_azure: bool = True
):
    """
    Inicializa o pipeline de COMEX.
    
    Args:
        iso_code: Código ISO do país (ex: 'ARG', 'BRA')
        start_date: Data inicial para coleta (padrão: 60 dias atrás)
        data_contract_path: Caminho para data-contract.yaml
        ids_table_path: Caminho para IDS_comex.xlsx
        storage_base_path: Caminho base no ADLS2 ou local
        use_azure: Se True, usa ADLS2; se False, usa storage local (OM/dados)
    """

Métodos Abstratos (devem ser implementados por cada país)

collect_import_data() → pd.DataFrame

Coleta dados brutos de importação específicos do país.

collect_export_data() → pd.DataFrame

Coleta dados brutos de exportação específicos do país.

_get_country_name() → str

Retorna nome do país para filtro na tabela de IDs.

_country_specific_treatment(df: pd.DataFrame) → pd.DataFrame

Aplica tratamento específico do país (opcional, retorna DataFrame sem modificação por padrão).

Métodos Comuns (implementados na classe base)

collect() → Tuple[pd.DataFrame, pd.DataFrame]

Executa fase de coleta completa, usando cache se disponível.

treat() → pd.DataFrame

Executa fase de tratamento: normalização, merge com IDs, validação.

calculate() → pd.DataFrame

Executa fase de cálculo: suavização, detecção de outliers, interpolação.

upload() → None

Executa fase de upload para API de Cost Drivers.

Storage e Cache

O framework suporta dois tipos de storage:

  • Azure ADLS2: Quando use_azure=True, salva em staging/comex/{ISO_CODE}/{camada}/
  • Local: Quando use_azure=False, salva em OM/dados/{ISO_CODE}/{camada}/

Além disso, há um sistema de cache local (apenas em desenvolvimento) que salva dados coletados em .cache/comex/{ISO_CODE}/ para evitar coletas repetidas durante desenvolvimento.

5. As 4 Fases do Pipeline

FASE 1: COLETA (collect)
↓
FASE 2: TRATAMENTO (treat)
↓
FASE 3: CÁLCULO (calculate)
↓
FASE 4: UPLOAD (upload)

Fase 1: Coleta (collect)

Objetivo: Coletar dados brutos de fontes externas (APIs, web scraping, arquivos).

Processo:

  1. Verifica cache local (se em ambiente de desenvolvimento)
  2. Se cache válido, usa dados do cache
  3. Senão, executa collect_import_data() e collect_export_data()
  4. Salva dados brutos em raw/import_raw_{data}.parquet e raw/export_raw_{data}.parquet
  5. Salva no cache local (se em desenvolvimento)

Saída:

  • OM/dados/{ISO_CODE}/raw/import_raw_{data}.parquet (local)
  • staging/comex/{ISO_CODE}/raw/import_raw_{data}.parquet (Azure)
  • OM/dados/{ISO_CODE}/raw/export_raw_{data}.parquet (local)
  • staging/comex/{ISO_CODE}/raw/export_raw_{data}.parquet (Azure)

Fase 2: Tratamento (treat)

Objetivo: Normalizar, validar e enriquecer os dados brutos.

Processo:

  1. Carrega dados brutos do storage (raw)
  2. Normaliza colunas usando data-contract.yaml (separadamente para import e export)
  3. Aplica tratamento específico do país (_country_specific_treatment())
  4. Padroniza tipos de dados conforme schema
  5. Concatena dados de importação e exportação
  6. Faz merge com IDS_comex.xlsx para obter IDIndicePrincipal
  7. Agrega por ID, Data e ImportExport
  8. Calcula FOB e CIF (quando disponível)

Saída:

  • OM/dados/{ISO_CODE}/silver/silver_{data}.parquet (local)
  • staging/comex/{ISO_CODE}/silver/silver_{data}.parquet (Azure)

Fase 3: Cálculo (calculate)

Objetivo: Aplicar cálculos estatísticos, suavização e detecção de outliers.

Processo:

  1. Carrega dados silver do storage
  2. Aplica suavização usando método de outliers (IQR) por grupo
  3. Detecta e trata outliers usando Q1, Q3 e IQR
  4. Calcula variação percentual mês a mês
  5. Preenche lacunas usando interpolação linear
  6. Obtém dados de cost drivers da API
  7. Faz merge com dados históricos

Saída:

  • OM/dados/{ISO_CODE}/gold/gold_{data}.parquet (local)
  • staging/comex/{ISO_CODE}/gold/gold_{data}.parquet (Azure)

Fase 4: Upload (upload)

Objetivo: Enviar dados finais para a API de Cost Drivers.

Processo:

  1. Carrega dados gold do storage
  2. Prepara dados no formato esperado pela API
  3. Valida dados antes de enviar
  4. Envia dados para API de Cost Drivers
  5. Registra resultados do upload

Saída:

  • Dados enviados para API de Cost Drivers
  • Logs de sucesso/erro do upload

6. Estrutura de Dados (OM/dados)

⚠️ Simulação da Saída dos Processos COMEX

O diretório OM/dados é onde estamos simulando a saída dos processos COMEX da metodologia antiga. Esta estrutura representa exatamente como os dados seriam organizados na saída dos pipelines de cada país.

Estrutura de Diretórios

OM/dados/
└── {ISO_CODE}/                    # Código ISO do país (ex: ARG, BRA, COL)
    ├── raw/                       # Camada Raw (Bronze) - Dados brutos
    │   ├── import_raw_{YYYY-MM-DD}.parquet
    │   └── export_raw_{YYYY-MM-DD}.parquet
    ├── silver/                    # Camada Silver - Dados tratados
    │   └── silver_{YYYY-MM-DD}.parquet
    └── gold/                      # Camada Gold - Dados finais
        └── gold_{YYYY-MM-DD}.parquet

Camada Raw (Bronze)

Conteúdo: Dados brutos coletados diretamente das fontes externas (APIs, web scraping, arquivos).

Formato: Parquet com colunas originais da fonte

Exemplo de arquivos:

  • OM/dados/BRA/raw/import_raw_2026-01-08.parquet
  • OM/dados/BRA/raw/export_raw_2026-01-08.parquet

Camada Silver

Conteúdo: Dados tratados, normalizados e validados. Colunas padronizadas, tipos corrigidos, merge com IDs de indicadores aplicado.

Formato: Parquet com schema padronizado

Colunas principais:

  • ID - ID do indicador (IDIndicePrincipal)
  • Data - Data do registro
  • FOB - Valor FOB
  • CIF - Valor CIF (quando disponível)
  • ImportExport - Tipo de comércio (0=export, 1=import)

Exemplo de arquivo:

  • OM/dados/BRA/silver/silver_2026-01-08.parquet

Camada Gold

Conteúdo: Dados finais com cálculos estatísticos aplicados. Suavização, detecção de outliers, interpolação, variação percentual, merge com dados históricos da API.

Formato: Parquet com dados prontos para upload

Colunas adicionais:

  • Colunas da camada Silver
  • Variacao - Variação percentual mês a mês
  • Dados suavizados e tratados
  • Dados históricos da API mergeados

Exemplo de arquivo:

  • OM/dados/BRA/gold/gold_2026-01-08.parquet

Quando os Dados são Salvos em OM/dados?

Os dados são salvos em OM/dados quando:

  • use_azure=False na inicialização do pipeline
  • O pipeline detecta que está rodando em ambiente local (não Airflow)
  • Azure Storage não está disponível (fallback automático)

Importante: Esta estrutura simula exatamente a saída dos processos COMEX da metodologia antiga, permitindo desenvolvimento e testes locais antes de fazer deploy para Azure ADLS2.

8. Execução no Airflow

DAGs Configurados

Os seguintes países possuem DAGs configurados no Airflow:

  • COMEX_ARG: COMEX_ARG_DAG.py
  • COMEX_BRA: COMEX_BRA_DAG.py
  • COMEX_COL: COMEX_COL_DAG.py
  • COMEX_DEU: COMEX_DEU_DAG.py
  • COMEX_EUR: COMEX_EUR_DAG.py
  • COMEX_USA: COMEX_USA_DAG.py

Estrutura do DAG

from airflow.decorators import dag, task
from airflow import utils
from datetime import datetime, timedelta

default_args = {
    'owner': 'airflow',
    'start_date': utils.dates.days_ago(2),
    'depends_on_past': False,
    'retries': 1,
    'retry_delay': timedelta(minutes=1)
}

@dag(
    dag_id="COMEX_BRA",
    default_args=default_args,
    schedule_interval=None,
    catchup=False,
    tags=['COMEX'],
    max_active_runs=1
)
def init():
    import sys
    import os
    airflow_include_path = os.getenv('AIRFLOW_INCLUDE_PATH', '/opt/airflow/include')
    sys.path.append(airflow_include_path)
    
    from COMEX_BRA import COMEX_BRA

    @task()
    def coleta():
        bot = COMEX_BRA()
        bot.collect()
    
    @task()
    def tratamento():
        bot = COMEX_BRA()
        bot.treat()
    
    @task()
    def calculo():
        bot = COMEX_BRA()
        bot.calculate()
    
    # Tasks chain
    step_coleta = coleta()
    step_tratamento = tratamento()
    step_calculo = calculo()
    
    step_coleta >> step_tratamento >> step_calculo

dag_def = init()

Fluxo de Execução

1. Task: coleta() - Executa bot.collect()
↓
2. Task: tratamento() - Executa bot.treat()
↓
3. Task: calculo() - Executa bot.calculate()

⚠️ Observação sobre Upload

Note que os DAGs atuais executam apenas as 3 primeiras fases (coleta, tratamento, cálculo). A fase de upload (bot.upload()) não está incluída nos DAGs por padrão, mas pode ser adicionada se necessário.

10. Troubleshooting

Problema: Dados não são salvos em OM/dados

Sintoma: Arquivos não aparecem em OM/dados/{ISO_CODE}/

Solução:

  1. Verifique se use_azure=False na inicialização
  2. Verifique se o diretório OM/dados/{ISO_CODE}/ existe
  3. Verifique permissões de escrita no diretório
  4. Verifique logs para erros de salvamento

Problema: Erro ao coletar dados

Sintoma: Erro durante fase de coleta

Solução:

  1. Verifique conexão com internet
  2. Verifique se a fonte de dados está acessível
  3. Verifique logs para erros específicos
  4. Tente usar cache local se disponível

Problema: Erro ao normalizar colunas

Sintoma: Erro durante fase de tratamento (normalização)

Solução:

  1. Verifique se data-contract.yaml existe e está correto
  2. Verifique se as colunas esperadas existem nos dados brutos
  3. Verifique se o mapeamento de colunas está correto para import e export
  4. Verifique logs para identificar coluna problemática

Problema: Cache não funciona

Sintoma: Cache não é usado mesmo em ambiente local

Solução:

  1. Verifique se está rodando em ambiente local (não Airflow)
  2. Verifique se o diretório .cache/comex/{ISO_CODE}/ existe
  3. Verifique se o cache não expirou (válido por 24 horas)
  4. Limpe o cache manualmente se necessário: rm -rf .cache/comex/{ISO_CODE}/

Dicas de Debug

  • Use logger.setLevel(logging.DEBUG) para ver logs detalhados
  • Inspecione DataFrames intermediários usando .head() e .info()
  • Verifique arquivos salvos em OM/dados/{ISO_CODE}/ após cada fase
  • Use cache local para evitar coletas repetidas durante desenvolvimento
  • Verifique se data-contract.yaml e IDS_comex.xlsx estão corretos
