1. Visão Geral

O que é o módulo serie_temporal?

O módulo serie_temporal implementa a Nova Metodologia Estatística para processamento de dados de comércio exterior. Ele faz parte da Nova Metodologia (NM) e utiliza técnicas de análise de séries temporais:

  • STL (Seasonal-Trend decomposition using Loess) para detecção de outliers
  • Top N Percent para agregação de dados por países relevantes
  • Cálculo de alpha (razão CIF/FOB) para valores CIF
  • Interpolação IQR para preenchimento de valores faltantes
  • Correção de valores negativos e ajustes no último mês

Objetivo Principal

Processar dados históricos de comércio exterior, aplicar metodologia estatística avançada para detectar e corrigir outliers, calcular valores FOB e CIF normalizados, e enviar para a API de Cost Drivers.

2. Nova Metodologia Estatística

Componentes da Nova Metodologia

A nova metodologia é composta por várias etapas estatísticas avançadas:

2.1. Top N Percent

Função: top_n_percent(df, threshold=0.8)

Descrição: Agrega dados por país dentro de cada grupo (IDIndicePrincipal, Data), ordena por peso decrescente, calcula valores acumulativos e identifica quais grupos atingem o threshold especificado (padrão: 80%).

Saída: DataFrame com colunas:

  • IDIndicePrincipal - ID do indicador
  • Data - Data do registro
  • FOB_100 - Preço médio FOB do grupo total (valor/peso)
  • FOB_80 - Preço médio FOB acumulativo (top 80%)
  • CIF_80 - Preço médio CIF acumulativo (top 80%)
  • pais_id_CUMUL - Lista acumulativa de códigos de países

2.2. STL (Seasonal-Trend decomposition using Loess)

Função: outlier_testing_stl(series, period=12, z_threshold=2.5)

Descrição: Decompõe a série temporal em três componentes: tendência, sazonalidade e resíduos. Identifica outliers usando z-score dos resíduos.

Componentes:

  • trend - Tendência da série
  • seasonality - Componente sazonal
  • residuals - Resíduos (diferença entre valor real e tendência+sazonalidade)
  • residuals_zscore - Z-score dos resíduos (usado para detectar outliers)

Outliers: Valores com abs(residuals_zscore) > z_threshold são marcados como outliers.

2.3. Interpolação IQR

Função: interpolation_iqr_stl(series, value_column='Valor_final')

Descrição: Preenche valores faltantes (NaN) usando múltiplas estratégias:

  1. Histórico preservado: Valores históricos (Type='historic') nunca são alterados
  2. Média móvel: Para NaN no início da série
  3. Tendência + Sazonalidade + Resíduo suavizado: Para NaN no meio da série
  4. Correção de outliers IQR: Se último mês é outlier por IQR, substitui por média móvel de 3 meses

2.4. Cálculo de Alpha

Função: find_alpha(series)

Descrição: Calcula a razão CIF/FOB (alpha) para cada IDIndicePrincipal e Data.

Fórmula: alpha = CIF_80 / FOB_80

Uso: Alpha é usado para calcular valores CIF quando apenas valores FOB estão disponíveis.

2.5. Correções Finais

Funções:

  • fix_last_month_high_residual() - Ajusta o último mês se for outlier
  • fix_negative_values() - Corrige valores negativos que podem ter sido gerados

Regras para último mês:

  1. Se outlier por resíduos E por IQR → média móvel 3 meses
  2. Se outlier por resíduos → tendência + sazonalidade + resíduos suavizados
  3. Se outlier por IQR → média móvel 3 meses

3. Arquitetura do Sistema

Padrão de Design

O módulo utiliza o padrão Template Method através de uma classe base abstrata (ComexPipelineNM) que define o fluxo comum em 4 fases, enquanto cada país implementa métodos específicos para suas particularidades.

Hierarquia de Classes

ComexPipelineNM (classe abstrata)
    ├── COMEX_BRA_NM (Brasil - coleta própria)
    ├── COMEX_AUT_NM (Áustria - usa EUR)
    ├── COMEX_ITA_NM (Itália - usa EUR)
    ├── COMEX_FRA_NM (França - usa EUR)
    ├── COMEX_BEL_NM (Bélgica - usa EUR)
    ├── COMEX_DEU_NM (Alemanha - usa EUR)
    ├── COMEX_PRT_NM (Portugal - usa EUR)
    ├── COMEX_NLD_NM (Holanda - usa EUR)
    ├── COMEX_HUN_NM (Hungria - usa EUR)
    └── COMEX_BGR_NM (Bulgária - usa EUR)

Métodos Abstratos (devem ser implementados)

collect_import_data() → pd.DataFrame

Coleta dados brutos de importação. Deve ser implementado quando iso_database != 'EUR'.

collect_export_data() → pd.DataFrame

Coleta dados brutos de exportação. Deve ser implementado quando iso_database != 'EUR'.

_get_country_name() → str

Retorna nome do país para filtro na tabela de IDs.

_country_specific_treatment(df) → pd.DataFrame

Aplica tratamento específico do país após normalização de colunas.

Métodos Comuns (implementados na classe base)

collect() → Tuple[pd.DataFrame, pd.DataFrame]

Fase 1: Coleta dados brutos de importação e exportação.

update_historical() → pd.DataFrame

Atualiza histórico com dados novos coletados.

normalize_historical() → pd.DataFrame

Fase 2: Normaliza colunas usando data-contract.yaml e aplica tratamento específico.

calculate() → pd.DataFrame

Fase 3: Aplica nova metodologia estatística (top N percent, STL, interpolação).

upload(ExcluirHistorico, percentualOutlier) → bool

Fase 4: Envia dados calculados para a API de Cost Drivers.

run(skip_phases) → pd.DataFrame

Método principal que orquestra todas as 4 fases.

5. Framework Base (costdrivers_comex_NM.py)

Classe ComexPipelineNM

A classe ComexPipelineNM é uma classe abstrata que define todo o fluxo comum de processamento em 4 fases. Ela utiliza o padrão Template Method, onde o fluxo geral é definido na classe base, mas métodos específicos são implementados pelas classes filhas.

Inicialização

def __init__(self, iso_code: str, start_date: Optional[datetime] = None, 
             data_contract_path: str = None, ids_table_path: str = None,
             storage_base_path: str = "staging/comex",
             use_azure: bool = True,
             threshold_percent: float = 0.8,
             min_months_required: int = 24,
             developing: bool = False,
             iso_database: Optional[str] = None):

Parâmetros Importantes:

  • iso_code - Código ISO do país (ex: 'BRA', 'AUT')
  • iso_database - Código ISO que indica origem dos dados (ex: 'EUR' para países europeus, que usam arquivos já separados por país em NM/dados/{ISO_CODE}/database/historical.parquet)
  • threshold_percent - Threshold para top_n_percent (padrão: 0.8 = 80%)
  • min_months_required - Mínimo de meses necessários para filtrar IDs (padrão: 24)
  • developing - Se True, não usa ADLS e salva/carrega tudo localmente

Armazenamento

O framework suporta dois modos de armazenamento:

  • Azure Data Lake Storage (ADLS2): Quando use_azure=True e developing=False
  • Armazenamento Local: Quando developing=True ou use_azure=False

Estrutura de Pastas no ADLS:

  • staging/comex/[ISO_CODE]/raw/ - Dados brutos coletados
  • staging/comex/[ISO_CODE]/historical/ - Dados históricos
  • staging/comex/[ISO_CODE]/normalized/ - Dados normalizados
  • staging/comex/[ISO_CODE]/calculated/ - Dados calculados

Cache Local

Em modo desenvolvimento (developing=True), o framework usa cache local:

  • Cache salvo em ./cache_[ISO_CODE]_import.parquet e ./cache_[ISO_CODE]_export.parquet
  • Cache válido por 24 horas
  • Útil para desenvolvimento e testes sem precisar coletar dados toda vez

  • Arquivos CSV com dados de importação e exportação por ano

  • Baixa arquivos CSV e processa

  • Processa múltiplos arquivos CSV
  • Normaliza colunas conforme data-contract.yaml
  • Concatena dados de importação e exportação

Países Europeus (AUT, ITA, FRA, BEL, DEU, PRT, NLD, HUN, BGR)

Fonte de Dados:

  • Dados já coletados e separados por país pelo processo OM/COMEX_EUR/COMEX_EUR.py
  • Arquivo específico de cada país: NM/dados/{ISO_CODE}/database/historical.parquet
  • Exemplos:
    • Áustria: NM/dados/AUT/database/historical.parquet
    • Itália: NM/dados/ITA/database/historical.parquet
    • França: NM/dados/FRA/database/historical.parquet
    • Alemanha: NM/dados/DEU/database/historical.parquet
    • E assim por diante para cada país...

Processamento:

  • Carrega histórico específico do país (já filtrado e separado)
  • Dados já foram filtrados por código ISO do país durante a coleta do EUR
  • Aplica tratamento específico do país
  • Cada país processa apenas seus próprios dados (não processa dados de outros países)

⚠️ Ponto de Atenção Importante:

Os arquivos históricos de cada país europeu devem ser atualizados pelo processo OM/COMEX_EUR/COMEX_EUR.py antes de executar os pipelines da Nova Metodologia. Este processo coleta os dados do EUR uma única vez e os separa por país, salvando em arquivos individuais. Isso evita processar os mesmos dados 9 vezes (uma vez para cada país europeu).

Fluxo correto:

  1. Executar OM/COMEX_EUR/COMEX_EUR.py para coletar e separar dados do EUR
  2. Este processo gera arquivos NM/dados/{ISO_CODE}/database/historical.parquet para cada país
  3. Executar pipelines individuais de cada país (COMEX_AUT_NM, COMEX_ITA_NM, etc.)
  4. Cada pipeline processa apenas seu próprio arquivo histórico

Tratamento Específico (comum a todos):

  • Filtra linhas onde NCM == "TOTAL"
  • Converte PERIOD (YYYYMM) para Data
  • Mapeia TRADE_TYPE para ImportExport
  • Remove coluna REPORTER
  • Define frete e seguro como NaN (não disponíveis nos dados EUR)

Detalhamento das Fases

Fase 1: Coleta

Para países com coleta própria (BRA):

  • Implementa collect_import_data() e collect_export_data()
  • Coleta dados diretamente da fonte (web scraping, APIs, etc.)
  • Salva dados brutos no storage
  • Atualiza histórico concatenando com dados anteriores

Para países europeus (iso_database='EUR'):

  • Fase de coleta é pulada
  • Dados são carregados diretamente do arquivo histórico específico do país: NM/dados/{ISO_CODE}/database/historical.parquet
  • Este arquivo é atualizado pelo processo OM/COMEX_EUR/COMEX_EUR.py
  • Cada país processa apenas seus próprios dados (já separados durante a coleta do EUR)

Fase 2: Normalização

Normaliza colunas usando data-contract.yaml:

  • Mapeia colunas brutas para colunas normalizadas
  • Converte tipos de dados
  • Aplica tratamento específico do país
  • Garante estrutura consistente para cálculo

Fase 3: Cálculo

Aplica nova metodologia estatística completa:

  1. find_IDs: Filtra apenas IDs com mínimo de 24 meses de dados
  2. top_n_percent: Agrega dados por top 80% de países por peso
  3. find_alpha: Calcula razão CIF/FOB para cada ID
  4. STL: Decompõe séries em tendência, sazonalidade e resíduos
  5. Detecção de outliers: Identifica valores anômalos por STL e IQR
  6. Interpolação: Preenche valores faltantes usando múltiplas estratégias
  7. Correções: Ajusta último mês e valores negativos
  8. Cálculo CIF: Calcula Valor_Cif usando alpha

Fase 4: Upload

Envia dados calculados para a API de Cost Drivers:

  • Prepara dados no formato esperado pela API
  • Obtém token de autenticação
  • Envia em batches para otimizar performance
  • Trata erros e retries

Arquivo data-contract.yaml

O que é o Contrato de Dados?

O arquivo data-contract.yaml especifica como as colunas brutas de cada país são mapeadas para as colunas normalizadas esperadas pelo pipeline. Este arquivo é fundamental para garantir que dados de diferentes fontes e formatos sejam processados corretamente.

Estrutura do Contrato

1. Schema (Definição de Tipos):

Define os tipos de dados esperados para cada coluna normalizada:

schema:
  Data: datetime
  ncm: int
  pais_id: int
  pais_name: str
  peso: float
  valor: float
  valor_cif: float
  frete: float
  seguro: float

2. Mapeamento por País:

Cada país possui uma seção com mapeamentos específicos para export e import:

[PAIS]:
  export:
    [COLUNA_BRUTA]: [COLUNA_NORMALIZADA]
    columns: [lista de colunas esperadas após normalização]
  import:
    [COLUNA_BRUTA]: [COLUNA_NORMALIZADA]
    columns: [lista de colunas esperadas após normalização]

Exemplos de Mapeamento por País

Brasil (BRA)

Características: Mesmas colunas para import e export, mas import tem colunas adicionais (frete e seguro)

BRA:
  export:
    CO_NCM: ncm              # NCM do Brasil
    CO_PAIS: pais_id         # Código do país
    KG_LIQUIDO: peso        # Peso líquido
    VL_FOB: valor           # Valor FOB
    columns: ['Data', 'ncm', 'ImportExport', 'pais_id', 'peso', 'valor']
  
  import:
    CO_NCM: ncm              # Mesma coluna para NCM
    CO_PAIS: pais_id         # Mesma coluna para país
    KG_LIQUIDO: peso        # Mesma coluna para peso
    VL_FOB: valor           # Mesma coluna para valor
    VL_FRETE: frete         # ⚠️ APENAS em importação
    VL_SEGURO: seguro       # ⚠️ APENAS em importação
    columns: ['Data', 'ncm', 'ImportExport', 'pais_id', 'peso', 'valor', 'frete', 'seguro']

União Europeia (EUR) e Países Europeus

Características: Usa nomenclatura diferente, com PRODUCT_NC para NCM

EUR:
  export:
    PRODUCT_NC: ncm          # ⚠️ Diferente do Brasil (CO_NCM)
    REPORTER: pais_origem    # País de origem
    PARTNER: pais_name       # Nome do país parceiro
    VALUE_EUR: valor         # Valor em EUR
    QUANTITY_KG: peso        # Quantidade em KG
    PERIOD: Data             # Período (YYYYMM)
    columns: ['Data', 'ncm', 'pais_origem', 'pais_name', 'ImportExport', 'peso', 'valor', 'frete', 'seguro']
  
  import:
    PRODUCT_NC: ncm          # Mesma coluna
    REPORTER: pais_origem   # Mesma coluna
    PARTNER: pais_name      # Mesma coluna
    VALUE_EUR: valor        # Mesma coluna
    QUANTITY_KG: peso       # Mesma coluna
    PERIOD: Data            # Mesma coluna
    columns: ['Data', 'ncm', 'pais_origem', 'pais_name', 'ImportExport', 'peso', 'valor', 'frete', 'seguro']

Colômbia (COL)

Características: ⚠️ COLUNAS DIFERENTES PARA IMPORT E EXPORT!

⚠️ ATENÇÃO: Na Colômbia, as colunas brutas são diferentes para importação e exportação!

COL:
  export:
    POSAR: ncm              # ⚠️ Coluna diferente para exportação
    FOBDOL: valor           # Valor FOB em dólares
    PBK: peso               # Peso
    columns: ['Data', 'ncm', 'ImportExport', 'peso', 'valor', 'frete', 'seguro']
  
  import:
    NABAN: ncm              # ⚠️ Coluna DIFERENTE para importação!
    VAFODO: valor           # Valor FOB em dólares (nome diferente)
    PBK: peso               # Mesma coluna para peso
    columns: ['Data', 'ncm', 'ImportExport', 'peso', 'valor', 'frete', 'seguro']

Diferenças Críticas:

  • Export: Usa POSAR para NCM
  • Import: Usa NABAN para NCM (coluna completamente diferente!)
  • Export: Usa FOBDOL para valor
  • Import: Usa VAFODO para valor (nome diferente)

Estados Unidos (USA)

Características: Colunas diferentes para import e export, import tem valor CIF

USA:
  export:
    E_COMMODITY: ncm        # ⚠️ Prefixo 'E_' para exportação
    CTY_CODE: pais_id       # Código do país
    CTY_NAME: pais_name     # Nome do país
    ALL_VAL_MO: valor       # Valor total
    QTY_1_MO: peso          # Quantidade
    time: Data              # Data
    columns: ['Data', 'ncm', 'ImportExport', 'pais_id', 'pais_name', 'peso', 'valor', 'frete', 'seguro']
  
  import:
    I_COMMODITY: ncm        # ⚠️ Prefixo 'I_' para importação (diferente!)
    CTY_CODE: pais_id       # Mesma coluna
    CTY_NAME: pais_name     # Mesma coluna
    GEN_VAL_MO: valor       # ⚠️ Nome diferente para valor
    GEN_CIF_MO: valor_cif   # ⚠️ Valor CIF (apenas em importação)
    GEN_QY1_MO: peso        # ⚠️ Nome diferente para peso
    time: Data              # Mesma coluna
    columns: ['Data', 'ncm', 'ImportExport', 'pais_id', 'pais_name', 'peso', 'valor', 'valor_cif']

Argentina (ARG)

Características: Colunas diferentes para país em import e export

ARG:
  export:
    ncm: ncm                # NCM já está normalizado
    pdes: pais_id           # ⚠️ 'pdes' para exportação
    pnet(kg): peso          # Peso líquido
    fob(usd): valor         # Valor FOB
    columns: ['Data', 'ncm', 'ImportExport', 'pais_id', 'peso', 'valor']
  
  import:
    ncm: ncm                # Mesma coluna
    porg: pais_id           # ⚠️ 'porg' para importação (diferente!)
    pnet(kg): peso          # Mesma coluna
    fob(usd): valor         # Mesma coluna
    flete(usd): frete       # Frete (apenas importação)
    seguro(usd): seguro     # Seguro (apenas importação)
    columns: ['Data', 'ncm', 'ImportExport', 'pais_id', 'peso', 'valor', 'frete', 'seguro']

⚠️ Pontos de Atenção Críticos

1. Diferenças entre Import e Export

MUITO IMPORTANTE: Alguns países têm colunas completamente diferentes para importação e exportação. Isso deve ser feito com muito cuidado ao configurar o contrato.

Exemplos de países com diferenças:

  • Colômbia: POSAR (export) vs NABAN (import) para NCM
  • Estados Unidos: E_COMMODITY (export) vs I_COMMODITY (import) para NCM
  • Argentina: pdes (export) vs porg (import) para país

Consequências de erro: Se você mapear incorretamente, os dados serão processados com valores errados ou colunas faltando, resultando em cálculos incorretos.

2. Colunas Obrigatórias vs Opcionais

Nem todos os países têm todas as colunas disponíveis:

  • Frete e Seguro: Geralmente disponíveis apenas em importação (ex: Brasil, Argentina)
  • Valor CIF: Disponível apenas em alguns países (ex: USA import tem GEN_CIF_MO)
  • País: Alguns países usam código (pais_id), outros usam nome (pais_name)

3. Como o Framework Usa o Contrato

Durante a fase de normalização (normalize_historical()), o framework:

  1. Carrega o arquivo data-contract.yaml
  2. Identifica o país sendo processado
  3. Identifica se os dados são de importação ou exportação
  4. Aplica o mapeamento correto conforme o tipo de comércio
  5. Renomeia colunas brutas para colunas normalizadas
  6. Valida que todas as colunas esperadas estão presentes

Exemplo de uso:

# Dados brutos do Brasil (importação)
df_raw = pd.DataFrame({
    'CO_NCM': [12345678],
    'CO_PAIS': [23],
    'KG_LIQUIDO': [1000.5],
    'VL_FOB': [50000.0],
    'VL_FRETE': [500.0],
    'VL_SEGURO': [100.0]
})

# Após normalização usando data-contract.yaml
df_normalized = pd.DataFrame({
    'ncm': [12345678],        # CO_NCM → ncm
    'pais_id': [23],          # CO_PAIS → pais_id
    'peso': [1000.5],         # KG_LIQUIDO → peso
    'valor': [50000.0],       # VL_FOB → valor
    'frete': [500.0],         # VL_FRETE → frete
    'seguro': [100.0]         # VL_SEGURO → seguro
})

Boas Práticas

  • Sempre verifique: Se as colunas brutas realmente existem nos dados antes de mapear
  • Teste separadamente: Import e export podem ter estruturas diferentes
  • Valide tipos: Certifique-se de que os tipos de dados estão corretos após mapeamento
  • Documente mudanças: Se adicionar um novo país, documente as particularidades
  • Use exemplos: Inclua exemplos de dados brutos e normalizados na documentação

schema:
  Data: datetime
  ncm: int
  pais_id: int
  pais_name: str
  peso: float
  valor: float
  valor_cif: float
  frete: float
  seguro: float

[PAIS]:
  export:
    [COLUNA_BRUTA_1]: [COLUNA_NORMALIZADA_1]
    [COLUNA_BRUTA_2]: [COLUNA_NORMALIZADA_2]
    ...
    columns: [lista de colunas esperadas após normalização]
  import:
    [COLUNA_BRUTA_1]: [COLUNA_NORMALIZADA_1]  # Pode ser diferente de export!
    [COLUNA_BRUTA_2]: [COLUNA_NORMALIZADA_2]
    ...
    columns: [lista de colunas esperadas após normalização]

Variáveis de Ambiente Necessárias

Variável Descrição Obrigatória
COSTDRIVERS_API_KEY Chave da API de Cost Drivers Sim
COSTDRIVERS_API_EMAIL Email para autenticação na API Sim
COSTDRIVERS_API_PASSWORD Senha da API Sim
AZURE_STORAGE_ACCOUNT_NAME Nome da conta Azure Storage (se use_azure=True) Condicional
AZURE_STORAGE_ACCOUNT_KEY Chave da conta Azure Storage (se use_azure=True) Condicional

Parâmetros de Inicialização

Parâmetro Descrição Padrão
iso_code Código ISO do país -
start_date Data inicial para coleta 60 dias atrás
threshold_percent Threshold para top_n_percent 0.8 (80%)
min_months_required Mínimo de meses para filtrar IDs 24
use_azure Usar Azure Storage True
developing Modo desenvolvimento (usa cache local) False
iso_database Código ISO que indica origem dos dados (ex: 'EUR' para países europeus) None (usa iso_code)

11. Troubleshooting

Problema: Erro ao carregar data-contract.yaml

Sintoma: FileNotFoundError: Arquivo data-contract.yaml não encontrado

Solução:

  1. Verifique se data-contract.yaml existe em NM/serie_temporal/
  2. Execute o script a partir da raiz do projeto
  3. Ou forneça o caminho absoluto: COMEX_BRA_NM(data_contract_path='/caminho/absoluto/data-contract.yaml')

Problema: Nenhum ID válido encontrado após find_IDs

Sintoma: Warning: Nenhum ID válido encontrado após find_IDs

Solução:

  1. Verifique se há dados históricos suficientes (mínimo 24 meses)
  2. Verifique se os NCMs nos dados correspondem aos NCMs em IDS_comex.xlsx
  3. Verifique se o nome do país em _get_country_name() corresponde ao nome em IDS_comex.xlsx
  4. Reduza min_months_required temporariamente para debug

Problema: Erro STL durante cálculo

Sintoma: ValueError: Error during STL decomposition

Solução:

  1. Verifique se há dados suficientes para decomposição STL (mínimo 2 períodos sazonais)
  2. Verifique se há valores NaN ou infinitos na série
  3. O erro é tratado automaticamente e a série original é retornada

Problema: Erro ao acessar Azure Storage

Sintoma: AzureException: Erro ao acessar ADLS2

Solução:

  1. Verifique se as variáveis de ambiente AZURE_STORAGE_ACCOUNT_NAME e AZURE_STORAGE_ACCOUNT_KEY estão configuradas
  2. Verifique se a conta de storage existe e está acessível
  3. Use developing=True para testar localmente sem Azure

Problema: Upload falhando

Sintoma: Status 401, 403 ou 500 nas requisições de upload

Solução:

  1. Verifique se o token está sendo obtido corretamente
  2. Verifique se as credenciais da API estão corretas
  3. Verifique se o formato dos dados está correto (tipos, colunas obrigatórias)
  4. Verifique se há valores NaN ou infinitos nos dados

Problema: Erro ao coletar dados do Brasil

Sintoma: RequestException: Erro ao acessar o portal COMEX

Solução:

  1. Verifique conexão com internet
  2. Verifique se o portal COMEX está acessível
  3. Verifique se a estrutura HTML do portal mudou (pode precisar atualizar código de scraping)
  4. Use cache local se disponível: developing=True

Dicas de Debug

  • Use developing=True para desenvolvimento local com cache
  • Use skip_phases para executar apenas fases específicas
  • Inspecione DataFrames intermediários usando .head() e .info()
  • Verifique logs detalhados usando logging.setLevel(logging.DEBUG)
  • Salve DataFrames intermediários em Parquet para inspeção: df.to_parquet('debug.parquet')
  • Para países europeus, verifique se o arquivo histórico específico do país existe: NM/dados/{ISO_CODE}/database/historical.parquet
  • Certifique-se de que o processo OM/COMEX_EUR/COMEX_EUR.py foi executado para atualizar os arquivos históricos
