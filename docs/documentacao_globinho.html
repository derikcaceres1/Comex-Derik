1. Visão Geral

O que é o Globinho?

O módulo globinho é responsável pelo processamento e cálculo (agrupamento) dos indicadores que possuem dado de transações por pais do comércio exterior. Ele faz parte da Nova Metodologia (NM) e utiliza uma arquitetura baseada em:

  • Polars para ingestão e transformação de dados (dados da europa tinha 290M linhas, teve que ser ajustado)
  • Pandas para operações finais e upload (compatibilidade com APIs)
  • Arquitetura orientada a objetos com classe base abstrata
  • Processamento assíncrono para uploads em lote

Objetivo Principal

Processar dados históricos de comércio exterior de diferentes países, normalizar formatos, fazer merge com indicadores, validar dados novos e enviar para a API de Cost Drivers.

Países Suportados

🇦🇷 Argentina (ARG)

Arquivo: ARG/COMEX_ARG_GLOBE.py

Processa dados de importação e exportação em formato Parquet.

🇧🇷 Brasil (BRA)

Arquivo: BRA/COMEX_BRA_GLOBE.py

Processa dados históricos do Brasil em formato Parquet.

🇺🇸 Estados Unidos (USA)

Arquivo: USA/COMEX_USA_GLOBE.py

Processa dados de importação e exportação dos EUA.

🇨🇴 Colômbia (COL)

Arquivo: COL/COMEX_COL_GLOBE.py

Processa dados de importação e exportação da Colômbia.

🇪🇺 União Europeia (EUR)

Arquivo: EUR/COMEX_EUR_GLOBE.py

Processa dados agregados da União Europeia por períodos.

2. Arquitetura do Sistema

Padrão de Design

O módulo utiliza o padrão Template Method através de uma classe base abstrata (BaseComexProcessor) que define o fluxo comum, enquanto cada país implementa métodos específicos para suas particularidades.

Hierarquia de Classes

BaseComexProcessor (classe abstrata)
    ├── COMEX_ARG_GLOBE (Argentina)
    ├── COMEX_BRA_GLOBE (Brasil)
    ├── COMEX_COL_GLOBE (Colômbia)
    ├── COMEX_EUR_GLOBE (União Europeia)
    └── COMEX_USA_GLOBE (Estados Unidos)

Métodos Abstratos (devem ser implementados)

load_raw_data() → pl.LazyFrame

Carrega dados brutos específicos do país usando Polars.

transform_data(lf: pl.LazyFrame) → pl.LazyFrame

Aplica transformações específicas do país (renomeação, conversões, filtros).

normalize_countries(df: pd.DataFrame) → pd.DataFrame

Normaliza IDs de países usando tabelas de mapeamento específicas.

Métodos Comuns (implementados na classe base)

load_indicator_ids(country_name: str) → pl.DataFrame

Carrega IDs de indicadores do arquivo IDS_comex.xlsx.

merge_with_indicators(lf, ids) → pl.LazyFrame

Faz merge dos dados com IDs de indicadores usando NCM e ImportExport.

group_data(lf) → pl.LazyFrame

Agrupa dados por dataNCM, idIndicador e idPais.

prepare_for_upload(df) → pd.DataFrame

Prepara DataFrame final com colunas padronizadas para upload.

validate_new_data(costdriver_df, new_df) → pd.DataFrame

Valida se há dados novos comparando com dados existentes na API.

run_upload(df) → List[Dict]

Executa upload assíncrono em batches para a API de Cost Drivers.

process() → List[Dict]

Método principal que orquestra todo o fluxo de processamento.

sql/funcao_airflow.py

NM/globinho/sql/funcao_airflow.py

Tipo: Função Airflow Task

Descrição: Função Airflow criada para contornar problemas de performance da API quando há grandes volumes de dados. Esta função:

  • Lê arquivo Parquet preparado (comex_europa_final.parquet) da pasta include do Airflow
  • Cria ou verifica existência da tabela TempInternationalTrade no SQL Server
  • Insere dados em batches de 100.000 registros para otimizar performance
  • Remove o arquivo Parquet após a carga bem-sucedida

Fluxo de Uso:

  1. Dados são processados e preparados localmente usando os processadores Python
  2. Arquivo Parquet final é gerado e manualmente enviado para o fileshare do Airflow na pasta include
  3. Função task_load_comex_parquet_to_sql() é executada no Airflow
  4. Após carga bem-sucedida, o script script_processamento_batch.sql é executado no banco de produção
⚠️ Requisitos:
  • Arquivo Parquet deve estar na pasta include do Airflow
  • Conexão com SQL Server configurada no Airflow (variáveis: server, database, username, password)
  • Usuário do banco deve ter permissões para INSERT e TRUNCATE

Estrutura do Arquivo Parquet Esperada:

  • Idindicador - ID do indicador
  • Datancm - Data do registro
  • Unid - Unidade
  • Importexport - Tipo de comércio (0=export, 1=import)
  • Idpais - ID do país
  • Uf - Unidade federativa
  • Via - Via de transporte
  • Quantidade - Quantidade
  • Peso - Peso líquido
  • Fob - Valor FOB
  • Frete - Valor do frete
  • Seguro - Valor do seguro
  • Valorcif - Valor CIF

sql/script_processamento_batch.sql

NM/globinho/sql/script_processamento_batch.sql

⚠️ Importante:

4. Pipeline Base (comex_globe_pipeline.py)

Classe BaseComexProcessor

A classe BaseComexProcessor é uma classe abstrata que define toda a lógica comum de processamento. Ela utiliza o padrão Template Method, onde o fluxo geral é definido na classe base, mas métodos específicos são implementados pelas classes filhas.

Inicialização

def __init__(self, config_path: str = 'config_comex.json'):
    """
    Inicializa o processador carregando configurações.
    
    Args:
        config_path: Caminho para o arquivo de configuração JSON
    """

Durante a inicialização:

  • Carrega arquivo de configuração JSON
  • Define tipos de colunas esperados
  • Configura endpoint da API
  • Inicializa logger

Busca Inteligente de Arquivos

O método _find_file() procura arquivos em múltiplos locais possíveis:

  1. Diretório atual
  2. Diretório NM/globinho/
  3. Diretório library/
  4. Raiz do projeto
  5. Subdiretórios de países dentro de globinho/
✓ Vantagem: Isso permite executar os scripts de qualquer diretório sem problemas de paths.

Fluxo de Processamento Completo

1. load_raw_data() - Carrega dados brutos (Polars)
↓
2. transform_data() - Transforma dados (Polars)
↓
3. load_indicator_ids() - Carrega IDs de indicadores
↓
4. merge_with_indicators() - Faz merge com IDs (Polars)
↓
5. group_data() - Agrupa por dataNCM, idIndicador, idPais (Polars)
↓
6. convert_to_pandas() - Converte para Pandas
↓
7. get_costdrivers_data() - Busca dados históricos da API
↓
8. validate_new_data() - Valida se há dados novos
↓
9. normalize_countries() - Normaliza IDs de países (Pandas)
↓
10. prepare_for_upload() - Prepara DataFrame final
↓
11. run_upload() - Upload assíncrono em batches

Validação de Dados Novos

Antes de fazer upload, o sistema valida se há dados novos comparando com dados existentes na API de Cost Drivers:

  1. Busca dados históricos dos últimos 3 meses da API
  2. Compara data máxima dos dados coletados com data máxima na plataforma
  3. Filtra apenas registros com data maior que a data máxima na plataforma
  4. Se não houver dados novos, o processamento é encerrado sem fazer upload

Upload Assíncrono

O upload é feito de forma assíncrona usando aiohttp e asyncio:

  • Batch Size: Configurável (padrão: 20 registros por batch)
  • Workers: Configurável (padrão: 10 requisições simultâneas)
  • Timeout: Configurável (padrão: 60 segundos)
  • Controle de Concorrência: Usa semáforo para limitar requisições simultâneas

5. Processadores por País

Argentina (COMEX_ARG_GLOBE)

Fonte de Dados:

  • NM/dados/ARG/import_hist_arg_raw.parquet - Dados de importação
  • NM/dados/ARG/export_hist_arg_raw.parquet - Dados de exportação

Transformações Específicas:

  • Unifica colunas com formatos diferentes (fob(u$s) vs fob(usd))
  • Converte tipos numéricos substituindo vírgulas por pontos
  • Filtra países nulos ou com código 999
  • Adiciona colunas frete e seguro como zero para exportação

Normalização de Países:

  • Usa paisID_ARG.csv para mapear IDs da fonte para IDs normalizados
  • Faz merge com tblNCM_Pais.xlsx usando nome do país

Brasil (COMEX_BRA_GLOBE)

Fonte de Dados:

  • NM/dados/BRA/database/historical.parquet - Dados históricos completos

Transformações Específicas:

  • Cria coluna data a partir de CO_ANO e CO_MES
  • Filtra apenas a data máxima disponível nos dados
  • Renomeia colunas: CO_NCM → ncm, CO_PAIS → pais, etc.

Normalização de Países:

  • Usa paisID_BRA.csv com encoding ISO-8859-1
  • Faz merge usando coluna CO_PAIS

Colômbia (COMEX_COL_GLOBE)

Fonte de Dados:

  • NM/dados/COL/import_hist_col_raw.parquet - Dados de importação
  • NM/dados/COL/export_hist_col_raw.parquet - Dados de exportação

Transformações Específicas:

  • Remove pontos e substitui vírgulas por pontos em valores numéricos
  • Filtra dados por formato de data (apenas YYYY válido)
  • Agrega dados e salva em formato particionado
  • Preenche NCM com zeros à esquerda até 10 dígitos

Normalização de Países:

  • Usa codpais_m49_colombia.xlsx (arquivo Excel)
  • Faz merge usando código M49 do país

União Europeia (COMEX_EUR_GLOBE)

Fonte de Dados:

  • NM/dados/EUR/database/historical.parquet - Dados agregados da UE

Transformações Específicas:

  • Processa dados por período (PERIOD)
  • Agrega por PERIOD, NCM, ImportExport e país
  • Salva dados agregados em formato particionado (PERIOD=YYYYMM)
  • Converte PERIOD (YYYYMM) para dataNCM (Date)

Normalização de Países:

  • Usa PARTNERS_ISO.txt para mapear códigos ISO
  • Faz merge usando código ISO do país parceiro

Estados Unidos (COMEX_USA_GLOBE)

Fonte de Dados:

  • NM/dados/USA/import_hist_usa_raw.parquet - Dados de importação
  • NM/dados/USA/export_hist_usa_raw.parquet - Dados de exportação

Transformações Específicas:

  • Agrega dados antes de processar

Normalização de Países:

  • Usa função normalize_country() para normalizar nomes
  • Faz merge usando nome normalizado do país

6. Fluxo Alternativo (Airflow + SQL)

Quando Usar o Fluxo Alternativo?

O fluxo alternativo usando Airflow e SQL foi criado para contornar problemas de performance da API quando há grandes volumes de dados. Este método é especialmente útil quando:

  • Há milhões de registros para upload
  • A API apresenta timeouts ou lentidão
  • É necessário garantir processamento batch mais eficiente
  • Há necessidade de controle mais granular sobre o processo de carga

Fluxo Completo do Processamento Alternativo

1. Processar dados localmente usando processadores Python
↓
2. Preparar DataFrame final com todas as colunas necessárias
↓
3. Salvar DataFrame como Parquet (ex: comex_europa_final.parquet)
↓
4. Enviar arquivo Parquet manualmente para fileshare do Airflow (pasta include)
↓
5. Executar funcao_airflow.py no Airflow (task_load_comex_parquet_to_sql)
↓
6. Função cria/verifica tabela TempInternationalTrade no SQL Server
↓
7. Função insere dados em batches de 100.000 registros
↓
8. Função remove arquivo Parquet após carga bem-sucedida
↓
9. Executar script_processamento_batch.sql no banco de produção
↓
10. Script processa dados e insere/atualiza tabelas finais

Preparação dos Dados

Antes de enviar para o Airflow, os dados devem estar no formato correto:

# Exemplo: Preparar dados para upload via Airflow
from NM.globinho.EUR.COMEX_EUR_GLOBE import COMEX_EUR_GLOBE
import pandas as pd

# Processar dados normalmente
processor = COMEX_EUR_GLOBE()
lf_raw = processor.load_raw_data()
lf_transformed = processor.transform_data(lf_raw)
ids = processor.load_indicator_ids("União Europeia")
lf_merged = processor.merge_with_indicators(lf_transformed, ids)
lf_grouped = processor.group_data(lf_merged)
df_pandas = processor.convert_to_pandas(lf_grouped)
df_normalized = processor.normalize_countries(df_pandas)
df_prepared = processor.prepare_for_upload(df_normalized)

# Renomear colunas para formato esperado pelo funcao_airflow.py
df_final = df_prepared.rename(columns={
    'idIndicador': 'Idindicador',
    'dataNCM': 'Datancm',
    'unid': 'Unid',
    'ie': 'Importexport',
    'idPais': 'Idpais',
    'uf': 'Uf',
    'via': 'Via',
    'quantidade': 'Quantidade',
    'peso': 'Peso',
    'fob': 'Fob',
    'frete': 'Frete',
    'seguro': 'Seguro',
    'valorCIF': 'Valorcif'
})

# Garantir todas as colunas esperadas
expected_columns = [
    'Idindicador', 'Datancm', 'Unid', 'Importexport', 'Idpais', 'Uf', 'Via',
    'Quantidade', 'Peso', 'Fob', 'Frete', 'Seguro', 'Valorcif'
]

for col in expected_columns:
    if col not in df_final.columns:
        df_final[col] = None

# Reordenar colunas
df_final = df_final[expected_columns]

# Salvar como Parquet
df_final.to_parquet('comex_europa_final.parquet', index=False, compression='gzip')

Configuração no Airflow

Variáveis necessárias no Airflow:

  • server - Servidor SQL Server (ex: "servidor.database.windows.net")
  • database - Nome do banco de dados
  • username - Usuário do banco de dados
  • password - Senha do banco de dados

Estrutura do DAG:

from airflow import DAG
from airflow.decorators import task
from airflow.models import Variable
from datetime import datetime
from funcao_airflow import task_load_comex_parquet_to_sql

# Definir variáveis de conexão (via Airflow Variables ou Connections)
server = Variable.get("sql_server")
database = Variable.get("sql_database")
username = Variable.get("sql_username")
password = Variable.get("sql_password")

@dag(
    dag_id="comex_load_parquet_to_sql",
    start_date=datetime(2024, 1, 1),
    schedule_interval=None,
    catchup=False
)
def comex_load_dag():
    load_task = task_load_comex_parquet_to_sql()
    
dag_instance = comex_load_dag()

Execução do Script SQL

⚠️ Importante: Após a execução bem-sucedida do funcao_airflow.py, o script script_processamento_batch.sql deve ser executado manualmente no banco de produção.

Requisitos:

  • Usuário do banco deve ter permissões para:
    • INSERT em cmx.tblSecexArquivo_CIF e cmx.tblSecexArquivo
    • TRUNCATE em TempInternationalTrade
    • SELECT nas tabelas de referência (tblIndicePrincipal, tblNCM_Pais)
  • Script deve ser executado no mesmo banco onde os dados foram carregados
  • Verificar se a tabela TempInternationalTrade contém dados antes de executar

Vantagens do Fluxo Alternativo

  • Performance: Inserção em batches de 100.000 registros é mais eficiente que upload via API
  • Confiabilidade: Menos dependência de timeouts e problemas de rede
  • Controle: Possibilidade de verificar dados antes de processar
  • Escalabilidade: Pode processar volumes muito maiores de dados
  • Auditoria: Dados ficam em tabela temporária antes do processamento final

Desvantagens do Fluxo Alternativo

  • Processo Manual: Requer upload manual do arquivo Parquet
  • Dependências: Requer acesso ao Airflow e ao banco de dados
  • Complexidade: Processo em duas etapas (Airflow + SQL)
  • Permissões: Requer usuário com permissões específicas no banco

8. Fluxo de Processamento Detalhado

Fase 1: Carregamento de Dados Brutos

Cada processador implementa load_raw_data() que retorna um pl.LazyFrame:

  • Argentina: Carrega dois arquivos Parquet (import/export) e concatena
  • Brasil: Carrega um único arquivo Parquet histórico
  • Colômbia: Carrega dois arquivos Parquet e agrega
  • União Europeia: Processa por períodos e agrega
  • Estados Unidos: Carrega dois arquivos Parquet e agrega

Fase 2: Transformação

transform_data() aplica transformações específicas usando Polars:

  • Renomeação de colunas para formato padrão
  • Conversão de tipos de dados
  • Criação de coluna de data (se necessário)
  • Filtros específicos do país
  • Renomeação final: data → dataNCM, pais → idPais

Fase 3: Merge com Indicadores

O sistema carrega IDs de indicadores do arquivo IDS_comex.xlsx e faz merge:

  • Filtra IDs pelo nome do país
  • Faz join usando ncm e ImportExport
  • Adiciona coluna idIndicador aos dados

Fase 4: Agrupamento

group_data() agrupa dados por:

  • dataNCM - Data do registro
  • idIndicador - ID do indicador
  • idPais - ID do país

E soma as colunas numéricas: peso, fob, frete, seguro

Fase 5: Validação de Dados Novos

Antes de fazer upload, o sistema valida se há dados novos:

  1. Busca dados históricos dos últimos 3 meses da API
  2. Encontra data máxima por indicador na plataforma
  3. Compara com data máxima dos dados coletados
  4. Filtra apenas registros com data maior que a data máxima na plataforma
  5. Se não houver dados novos, retorna DataFrame vazio e encerra

Fase 6: Normalização de Países

Cada país implementa normalize_countries() que:

  • Carrega tabela de mapeamento específica do país
  • Faz merge com tblNCM_Pais.xlsx
  • Substitui idPais pelo ID normalizado

Fase 7: Preparação para Upload

prepare_for_upload() prepara o DataFrame final:

  • Garante todas as colunas esperadas pela API
  • Preenche valores faltantes com NaN ou valores padrão
  • Converte tipos de dados conforme esperado
  • Formata data como string (YYYY-MM-DD)

Fase 8: Upload Assíncrono

run_upload() executa upload em batches:

  • Gera batches agrupados por idIndicador
  • Envia requisições assíncronas usando aiohttp
  • Controla concorrência com semáforo
  • Retorna lista com resultados de cada batch (status, resposta)

10. Troubleshooting

Problema: Arquivo de configuração não encontrado

Sintoma: FileNotFoundError: Arquivo de configuração não encontrado

Solução:

  1. Verifique se config_comex.json existe em NM/globinho/
  2. Execute o script a partir da raiz do projeto
  3. Ou forneça o caminho absoluto: COMEX_ARG_GLOBE(config_path='/caminho/absoluto/config.json')

Problema: Variáveis de ambiente não configuradas

Sintoma: ValueError: Variável de ambiente COSTDRIVERS_API_KEY não encontrada

Solução:

  1. Crie arquivo .env na raiz do projeto
  2. Adicione as variáveis: COSTDRIVERS_API_KEY, COSTDRIVERS_API_EMAIL, COSTDRIVERS_API_PASSWORD
  3. Ou configure como variáveis de ambiente do sistema

Problema: Arquivo de dados não encontrado

Sintoma: FileNotFoundError: Arquivo de importação não encontrado

Solução:

  1. Verifique se os arquivos Parquet existem nos caminhos especificados em config_comex.json
  2. Verifique os paths em config['paths'][country_code]
  3. O sistema tenta encontrar em múltiplos locais, mas os arquivos devem existir

Problema: Nenhum ID encontrado para o país

Sintoma: Warning: Nenhum ID encontrado para país 'X'

Solução:

  1. Verifique se o nome do país em get_country_name() corresponde ao nome em IDS_comex.xlsx
  2. Verifique se há dados para o país na coluna Pais_1 do arquivo Excel
  3. O nome deve corresponder exatamente (case-sensitive)

Problema: Erro no merge com indicadores

Sintoma: DataFrame vazio após merge ou erro de tipo

Solução:

  1. Verifique se os tipos de ncm e ImportExport são compatíveis
  2. Verifique se os valores de NCM nos dados correspondem aos valores em IDS_comex.xlsx
  3. Verifique se ImportExport está no formato correto (0 ou 1)

Problema: Upload falhando

Sintoma: Status 401, 403 ou 500 nas requisições de upload

Solução:

  1. Verifique se o token está sendo obtido corretamente
  2. Verifique se as credenciais da API estão corretas
  3. Verifique se o formato dos dados está correto (tipos, colunas obrigatórias)
  4. Reduza batch_size e workers se houver muitos erros

Problema: Nenhum dado novo para upload

Sintoma: Mensagem "Nenhum dado novo para upload"

Explicação: Isso não é um erro! O sistema validou que não há dados novos comparando com a plataforma.

Para forçar upload: Modifique temporariamente validate_new_data() ou ajuste a lógica de comparação de datas.

Dicas de Debug

  • Use logger.setLevel(logging.DEBUG) para ver logs detalhados
  • Inspecione DataFrames intermediários usando .head() e .info()
  • Verifique schemas de LazyFrames usando .collect_schema()
  • Salve DataFrames intermediários em Parquet para inspeção: df.to_parquet('debug.parquet')
  • Execute cada fase separadamente para identificar onde ocorre o problema
